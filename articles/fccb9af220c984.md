## 動機

[moonbit-community/isomorphic](https://github.com/moonbit-community/isomorphic)という、フロントエンドとバックエンドの両方がMoonBitで書かれたサンプルアプリケーション集があります。

https://github.com/moonbit-community/isomorphic

コードを読んでいると、サーバーサイドのJSONのデコードやバリデーションを逐次行っており、実際に開発するときのハードルになりそうでした。

..post(@shared.api_contacts, event => {
  println("POST \{@shared.api_contacts}")

## 前提

### フレームワーク

`isomorphic`リポジトリのコードは、フロントエンドは[rabbita](https://github.com/moonbit-community/rabbita)、バックエンドは[oboard/mocket](https://github.com/oboard/mocket)というフレームワークを使っています。

`rabbita`はThe Elm Architecture風のUIフレームワークで、MoonBitのパッケージレジストリである[mooncakes.io](https://mooncakes.io/)で使われています。

https://moonbit-community.github.io/rabbita/

`mocket`については、今回のRPC移植にあたって[mizchi/mars.mbt](https://github.com/mizchi/mars.mbt)に書き換えたので、説明を省略します。

### MoonBitでのJSONのデコード

MoonBitには[組み込みのJson型](https://docs.moonbitlang.com/en/stable/language/fundamentals.html#json)があります。冒頭のコードでは、`Json`に対してパターンマッチをして安全にデコードしていました。

デコードの例:

```mbt check
struct User {
  id : Int
  name : String
} derive(Show, Eq)

test "Json" {
  fn decode_user_json(json : Json) -> User? {
    let id = match json {
      { "id": Number(id, ..), .. } => id
      _ => return None
    }
    let name = match json {
      { "name": String(name), .. } => name
      _ => return None
    }
    Some({ id: id.to_int(), name })
  }

  assert_eq(decode_user_json({ "id": 1, "name": "me" }),Some({ id: 1, name: "me" }))
  assert_eq(decode_user_json({ "id": "invalid", "name": "me" }), None)
}
```

また、`FromJson`、`ToJson`トレイトを使ってエンコード・デコードすることもできます。

https://docs.moonbitlang.com/en/latest/language/derive.html#fromjson-and-tojson

```mbt check
struct User2 {
  id : Int
  name : String
} derive(Show, Eq, FromJson, ToJson)

test "Json trait" {
  fn decode_user_json(json : Json) -> User2? {
    let user : User2 = @json.from_json(json) catch {
      JsonDecodeError(_) => return None
    }
    Some(user)
  }

  assert_eq(decode_user_json({ "id": 1, "name": "me" }), Some({ id: 1, name: "me" }))
  assert_eq(decode_user_json({ "id": "invalid", "name": "me" }), None)
}
```

RPCとして普通に使う場合はデコードに失敗することはないので、トレイトを使ってデコードすることにします。

## 作ったもの

https://github.com/tekihei2317/moonbit-rpc-poc

[tRPC](https://trpc.io/)風にしてみました。RPCの仕様はtRPCと同じく次の通りです。

| HTTPメソッド | 操作 | 備考                                                                                                              |
| ------------ | ------------ | ----------------------------------------------------------------------------------------------------------------- |
| GET          | query        | inputをJSONエンコードしてクエリパラメータに入れる<br>`myQuery?input=${encodeURIComponent(JSON.stringify(input))}` |
| POST         | mutation     | inputをリクエストボディに入れる                                                                                   |

[HTTP RPC Specification | tRPC](https://trpc.io/docs/rpc)

### 型安全なRPCの実現方法

TypeScriptのように、オブジェクトから柔軟に型を導くことは難しいです。そのため、エンドポイントとその入出力を表す次のような`Route`型を作成し、クライアントとサーバーで共有します。

```mbt
///|
/// The RPC route kind.
pub enum Operation {
  Query
  Mutation
}

///|
/// Shared route description used by both client and server.
pub struct Route[Input, Output] {
  operation : Operation
  name : String
}
```

クライアントとサーバーでは、それぞれ`@rpc.call`と`@rpc.handle`を使って呼び出しと実行を記述します。ここでは`rabbita`と`mizchi/mars`に接続しています。

https://github.com/tekihei2317/moonbit-rpc-poc/blob/22069dc53724d7afdd6951652d7b6bbe114e6e11/frontend/rpc/client.mbt#L15-L39

https://github.com/tekihei2317/moonbit-rpc-poc/blob/main/backend/rpc/server.mbt#L49-L100


## 使用例

`moonbit-community/isomorphic`の連絡先アプリを、RPCで書き直してみました。

### ルートの定義

入出力は構造体で定義します。`query`ヘルパと`mutation`ヘルパで`Route`を定義します。

```mbt
///|
pub(all) struct CreateContactInput {
  name : String
  email : String
  phone : String
} derive(ToJson, FromJson)

///|
pub(all) struct ContactIdInput {
  id : Int
} derive(ToJson, FromJson)

///|
/// 連絡先の一覧の取得
pub let list_contacts : Route[Unit, Array[Contact]] = query("contacts.list")

///|
/// 連絡先の登録
pub let create_contact : Route[CreateContactInput, Contact] = mutation(
  "contacts.create",
)

///|
/// 連絡先のお気に入りの切り替え
pub let toggle_contact_favorite : Route[ContactIdInput, Contact] = mutation(
  "contacts.toggle_favorite",
)

///|
/// 連絡先の削除
pub let delete_contact : Route[ContactIdInput, Unit] = mutation(
  "contacts.delete",
)
```

### クライアント側

`dispatch`は指定したメッセージで`update`を呼び出す`rabbita`の関数です。

```mbt
/// 連絡先の一覧を取得する
@rpc.call(@shared.list_contacts, (), result => dispatch(GotContacts(result)))

// 連絡先を登録する
@rpc.call(
  @shared.create_contact,
  { name, email, phone },
  result => dispatch(ContactAdded(result)),
)
```

### サーバー側

`get_all_contacts`と`create_contact`はデータベースとやり取りする関数です。

```mbt
/// 連絡先の一覧を取得する
@rpc.handle(app, @shared.list_contacts, _ => {
  let contacts = get_all_contacts(conn) catch {
    _ => return Err(@shared.RpcError::internal())
  }
  Ok(contacts)
})

// 連絡先を登録する
@rpc.handle(app, @shared.create_contact, input => {
  // バリデーションのコードは省略
  let contact = create_contact(conn, input.name, input.email, input.phone) catch {
    _ => return Err(@shared.RpcError::internal())
  }
  Ok(contact)
})
```

## ハマったところ

`isomorphic`はMoonBit nightly前提で動かす前提で作られているのですが、nightlyの言語サーバーがうまく動かなかったのでstableを使うことにしました。その際、次のようにモジュール構成を書き換える必要がありました。

nightly: 単一のモジュール

```text
- backend/
  - moon.pkg
- frontend/
  - moon.pkg
- shared/
  - moon.pkg
- moon.mod.json
```

stable: 複数のモジュール

```
- backend/
  - moon.pkg
  - moon.mod.json
- frontend/
  - moon.pkg
  - moon.mod.json
- shared/
  - moon.pkg
  - moon.mod.json
```

モジュールを複数作っている理由は、stableの言語サーバーが`moon.mod.json`の`preferred_target`を使用するからです[^1]。単一のモジュール構成にすると`js`ターゲットと`native`ターゲットのどちらかのインポートでエラーになってしまいました。

[^1]: https://github.com/moonbitlang/moonbit-docs/blob/2efb6a95c85b7117e9a6110e2fc45448c216b6e3/next/toolchain/moon/module.md#L130-L134

## おわりに

TypeScriptの経験があれば割と自然に書けると思うので、是非気軽に試してみて欲しいです。LSPの体験がいいところ、パイプラインが使えること、エラーハンドリングが強制されること、docテストが組み込まれているところなどが好きです。

Rustや関数型言語の経験がある方にとってはあまり新鮮味がないかもしれません（前向きに言い換えれば、すぐ使えるようになるということです）。

本記事で作ったグルーコードは稚拙ですが、Next.jsのようなフルスタックフレームワークが登場すればMoonBitは広く使われるようになるのではないのかなという気がしています。mizchiさんの[mizchi/sol.mbt](https://github.com/mizchi/sol.mbt)にも期待です。
